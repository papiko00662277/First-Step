# 要件定義書：「最初の一歩」タスク分解アプリ（仮称 FirstStep）

作成日：2026-06-22
作成者：tomo

---

## 1. 背景・課題

やるべきことは分かっているのに、なかなか着手できず先延ばししてしまう。この原因の多くは「タスクが大きい・曖昧で、机に向かって"最初に何をすればいいか"が分からない」ことにある。

既存のToDoアプリは「タスクを記録する」ことはできても、「どう始めるか」までは助けてくれない。一方、生成AIに「このタスクを分解して」と単純に頼むと、「計画を立てる」「情報を集める」といった抽象的で一般論的なステップが返ってきて、結局その場で動けない。

つまり、**「曖昧で大きいタスク」と「今すぐ動ける具体行動」の間にあるギャップ**を埋める仕組みが必要。

## 2. 目的・ゴール

曖昧で大きなタスクを、**今すぐ2分以内で着手できる超具体的な「最初の一歩」**に変換する。これにより着手のハードルを下げ、先延ばしを減らす。

「自分の生活を豊かにする」という観点では、先延ばしによる心理的負担の軽減と、小さな着手による達成感の積み重ねを狙う。

### 成功の定義（KPI 案）
- 提示された「最初の一歩」が、追加で考えずにそのまま行動に移せる割合
- 提示後に実際に着手できたタスクの割合（自己申告でも可）

## 3. ターゲットユーザー

自分自身（大学院生）。研究・授業課題・事務手続きなどで先延ばししがちな場面を想定。

### 想定シーン
- 「修論の章を書く」としか思いつかず、机に座っても手が止まる
- メール返信や書類提出など、面倒で後回しにしているタスクが溜まっている

## 4. スコープ

### 対象（In Scope / MVP）
- タスクの入力
- タスクの分解（粒度を制約）
- 「最初の一歩」の抽出と提示
- 各ステップの所要時間目安の提示
- （任意）状況の補足入力による出力の最適化

### 対象外（Out of Scope / MVP）
- 本格的なタスク管理（期限管理、リマインド通知、繰り返し設定）
- 複数ユーザー・共有機能
- データの永続保存・クラウド同期（MVPはセッション内のみ）

## 5. ユーザーストーリー

- user として、「レポートを書く」しか思いつかないとき、**まず机で最初にやる一手**を具体的に教えてほしい。動き出すきっかけが欲しいから。
- user として、分解されたステップが抽象的だと動けないので、**「動詞＋具体物」で2分以内に始められる粒度**で示してほしい。
- user として、締切やつまずいている点を伝えたら、**それを踏まえた**最初の一歩にしてほしい。

## 6. 機能要件

| ID | 機能 | 内容 |
|----|------|------|
| F1 | タスク入力 | 自由記述でタスクを入力できる |
| F2 | 文脈の補足（任意） | 締切・現状の進捗・つまずいている点を任意で入力できる |
| F3 | タスク分解 | タスクを実行可能なステップ列に分解する（粒度制約あり） |
| F4 | 最初の一歩の抽出 | 「2分以内で着手可能・具体的」な最初の一手を1つ強調表示する |
| F5 | 自己検証 | 各ステップが「具体的か」「2分で着手可能か」をAI自身が点検・修正してから出力する |
| F6 | 着手後の次の一手（任意） | 着手できたら次のステップを提示する |
| F7 | ペナルティ／コミットメント機構 | 「いつまでに最初の一歩をやる」と宣言し、達成できなかった場合の自己設定ペナルティを登録。期限経過時に未達なら通知し、損失回避を着手の後押しに使う |

## 7. 非機能要件

- **応答性**：入力から数秒〜十数秒で結果が返る
- **わかりやすさ**：最初の一歩が一目で分かるUI（強調表示）
- **可搬性**：単一ファイル（HTML/JS）で完結し、提出・デモが容易
- **再現性**：同じ入力に対して品質が安定する（プロンプトで担保）

## 8. 入出力仕様

### 入力
- 必須：タスク文（自由記述）
- 任意：締切、現状の進捗、つまずいているポイント

### 出力
- **最初の一歩**：1つ、超具体（動詞＋具体物、2分以内で着手可能）
- 続くステップ：3〜5個、各ステップに所要時間の目安
- （任意）着手を後押しする一言

## 9. 工夫の方針（評価対象）

### 9-1. プロンプト設計の工夫 ―「単純プロンプトでは失敗した → だから○○を導入」

| 観察された失敗 | 原因 | 導入する工夫 |
|----------------|------|--------------|
| ステップが抽象的（「計画を立てる」） | 役割・粒度の指定がない | **役割付与**（先延ばし支援コーチ）＋**粒度制約**（「動詞＋具体物＋2分以内」） |
| 一般論で自分の状況に合わない | 出力の良し悪しの基準が曖昧 | **Few-shot**（良い例／悪い例を提示） |
| いきなり結論を出して粗い | 思考の段取りがない | **段階分解**（内部で分解→検証→出力の手順を踏ませる） |
| ユーザーの状況を無視 | 文脈情報が渡っていない | **文脈質問の導入**（F2）と、その値のプロンプトへの差し込み |
| たまに2分で始められない一手を出す | 出力後の自己点検がない | **自己検証ステップ**（self-check）で再評価・修正させる |

### 9-2. 設計プロセスの工夫 ―「なぜその手順か」を説明できるように

- **なぜこの手順か**：要件定義 → プロンプト試作 → 失敗の観察（実タスクで検証） → 改善の反復 → UI実装、という順で進める。先にプロンプトの良し悪しを確かめてからUIに固めることで、作り直しを減らす。
- **なぜこの構成か**：分解結果を全部見せるより「最初の一歩」を1つ強調する方が着手率が上がるという仮説に基づき、出力とUIの主役を「最初の一歩」に置く。

## 10. 実装形態（案）

Webアプリ（HTML/JS 単体）。ブラウザで動き、1ファイルで提出・デモできる形を第一候補とする。生成AIの呼び出し方法（API利用 / 手元での実行 / デモ用のモック）は、利用環境に合わせて次フェーズで確定する。

## 12. 理論的根拠（先行研究レビュー）

本アプリのコンセプト「**最初の一歩（最初の数分）を具体化することが、着手と継続のカギになる**」を、行動変容・HCI 分野の先行研究で裏づける。

**採録基準**：本プロジェクトで参照する論文は、すべて **CORE Rank A 以上**（A／A*）の会議に採録されたものに限定し、**2つのテーマ（①分解・着手、②ペナルティ）ごとに2本ずつ**、できるだけ新しいものを優先して採用する。CORE は計算機科学系の会議を対象とするランキングのため、本レビューでは行動変容・HCI のトップ会議（CHI＝A*、UbiComp/IMWUT＝A*）の実証研究を採用する。心理学・経済学のジャーナル論文（実行意図のメタ分析や習慣形成の古典など）は概念的背景としては有名だが CORE の対象外のため、参照文献からは外している。

### テーマ①：曖昧なタスクを「分解」して「最初の一歩」で着手させる

#### 12-1. 大きなタスクは「分解」した方が、質が上がり中断にも強い ＜CHI 2015 / A*＞
Cheng, Teevan, Iqbal & Bernstein (2015, CHI) は、まとまった作業（マクロタスク）を小さな単位（マイクロタスク）に分解する効果を実験的に検証し、分解すると**成果物の質が上がり、中断からの復帰に強く、体験も良くなる**ことを示した。

→ **アプリへの反映**：曖昧で大きいタスクをステップに分解する中核機能（F3）と、特に最初の単位を最小化する設計（F4）の根拠。

#### 12-2. 生成AIが「停滞」を検知し、続きを見せて再開のハードルを下げる ＜CHI 2023 / A*＞（最新）
Arakawa, Yakura & Goto (2023, CHI) の CatAlyst は、作業の停滞を検知し、**大規模生成モデルが「作業の続き」を生成して提示する**ことで、先延ばしを防ぎ再開のハードルを下げる介入を提案・評価した。成果物を肩代わりするのではなく、関心を引いて着手・再開を促す点が要点。

→ **アプリへの反映**：生成AIで「最初の一歩」を具体的に示して着手のハードルを下げる本アプリの中核アプローチ（F4／F6）を、最新のHCI研究が直接裏づける。

### テーマ②：ペナルティ（コミットメント）で着手・継続を強める

#### 12-3. 金銭的コミットメント（ペナルティ）は達成率を上げる ＜CHI 2021 / A*＞（最新）
Lee, Kim et al. (2021, CHI) の Sticky Goals は、コミットメント支援サービス stickK の大規模データを分析し、**未達時に掛け金が没収される金銭的コミットメントを設定した目標は、設定しない目標より達成率が有意に高い**こと、金銭的・社会的コミットメントの組み合わせ方が達成を左右することを示した。

→ **アプリへの反映（F7）**：「いつまでに最初の一歩をやる」と宣言させ、未達時の自己設定ペナルティを登録できるようにする根拠。損失回避という強い動機を着手の引き金として重ねる。

#### 12-4. 自分で課す穏当なペナルティは「自己制御の道具」として機能する ＜IMWUT/UbiComp 2019 / A*＞
Kim, Jung, Ko & Lee (2019, IMWUT) の GoalKeeper は、コミットメント・デバイスを「目標達成までの自発的な制限、または未達時に自発的に課す**ペナルティ**」と定義し、スマホ使用の自己制御における各種ロックアウト機構の強度と遵守の関係を検証。**ユーザーが自分でペナルティ強度を選び、戦略的に使う**ことで自己制御に役立つことを示した。

→ **アプリへの反映（F7）／倫理的配慮**：ペナルティは**ユーザー自身が設定する穏当なもの**（例：寄付、ご褒美の先送り、SNS報告）に限り、強度も本人が選べるようにする。自傷・健康を害する内容や過度に重い罰は提案・許容しない。あくまで前向きな後押しとして機能させる。

### 注記：「2分ルール」など実務フレームワークとの関係
着想のもとになった「2分ルール」は David Allen『Getting Things Done』や James Clear『Atomic Habits』で広まった**実務フレームワーク**であり、それ自体は査読論文ではない。本書では効果の根拠を上記の CORE A 以上の実証研究に置き、実務ルールは説明の補助としてのみ扱う。

## 15. 競合・類似アプリ調査と差別化方針

GitHub・各種プロダクトを調査した結果、類似アプリは大きく**2系統**に分かれ、本アプリが狙う「分解」と「ペナルティ」を**1つのループに統合した例は見当たらなかった**。

### 15-1. 系統A：AIで「最初の一歩」に分解する着手支援系

| アプリ | 概要 | 強み | 弱み（＝本アプリの余地） |
|--------|------|------|--------------------------|
| **Thawly.ai** | 認知心理学研究者（ADHD当事者）が開発。タスクを「最小の身体的マイクロ行動」へ即分解し、**2分タイマー**で勢いをつける | 着想が本アプリと最も近い。最小行動＋2分の組み合わせ | ペナルティ/コミットメント機構なし。継続の強制力が弱い |
| **ADHD Guardian**（OSS, hackathon） | GPT-4oでタスクを**3〜5個の最初の一歩**に分解＋**非批判的な励まし**（ペルソナ"Em"）。Flask/HTML/Tailwind、Graph API連携（予定） | オープンソースで実装参照可。励ましの人格設計 | ペナルティなし。状況（締切・つまずき）への適応が浅い。デモ段階 |
| **adhd-tasker**（OSS） | 「今やる1タスクだけ全画面表示」。曖昧さを消すUI | 集中UIが明快 | AI分解なし・ペナルティなし |
| Shrimp Task Manager / Conductor Tasks / Taskosaur（OSS） | AIエージェント/開発者向けのタスク自動分解 | 依存関係管理・構造化が強力 | 対象が**ソフトウェア開発**で、個人の生活・着手心理は対象外 |

→ **把握**：系統Aは「分解して最初の一歩を示す」までは本アプリと重なる。Thawly は2分＋マイクロ行動まで実装済みで、**「2分」「分解」だけでは差別化にならない**。一方、いずれも**未達時の強制力（ペナルティ）を持たない**。

### 15-2. 系統B：お金を賭けるペナルティ/コミットメント系

| アプリ | 仕組み | 数値 | 弱み（＝本アプリの余地） |
|--------|--------|------|--------------------------|
| **stickK** | 目標未達でカード課金→慈善/アンチ慈善/友人へ。完全無料 | 賭けあり78% vs なし35%の達成率 | タスクは**自分で用意**前提。分解・着手支援なし |
| **Beeminder** | "Bright Red Line"を割ると$5→$10→…→$810と段階課金。多数の自動計測連携 | データ計測志向 | 同上。着手の最初の一歩は助けない |
| **Forfeit**（YC出資） | 写真/タイムラプス/Apple Health/GPS/RescueTime/自己申告で証拠提出、GPTが検証。寛容度モード・appeal機能 | 累計**$8.7M**賭け・成功率**94.2%** | **習慣丸ごとの完了**に賭ける設計。分解・着手支援なし。罰金は基本アプリ取得 |

→ **把握**：系統Bは強い継続力を持つが、**「何をやるか」が既に分かっている前提**で、**大きく曖昧なタスクを最初の一歩に砕く部分を助けない**。賭けの単位も「習慣/タスク丸ごと」で、着手のハードル自体は下げない。

### 15-3. 空白地帯と本アプリの独自化方針

調査から、**「AIによる最初の一歩への分解」と「ペナルティによる着手の強制」を一つの行動ループに統合した製品は存在しない**ことが分かった。ここを本アプリの独自ポジションとする。

1. **「最初の一歩」へのマイクロ・ペナルティ（中核の新規性）**
   既存のペナルティ系は「習慣丸ごと」に賭ける。本アプリは**AIが切り出した2分の最初の一歩**に少額・穏当なペナルティをかける。賭ける対象が「2分で着手可能な行動」なので、心理的にも金銭的にも軽く、着手のハードルを下げつつ損失回避を効かせられる（系統A＋Bの“いいとこ取り”）。根拠：CatAlyst／Break It Down（分解・着手）＋ Sticky Goals／GoalKeeper（ペナルティ）。
2. **AIの自己検証ループ**：各ステップが「動詞＋具体物・2分で着手可能か」をAI自身が点検・修正してから提示（F5）。Thawly/ADHD Guardian は分解の品質保証を明示しておらず、ここはプロンプト設計の工夫として差別化＝授業評価軸に直結。
3. **穏当・自己設定ペナルティ＋強度選択**：金銭に限らず寄付・ご褒美先送り・SNS報告などを本人が選び、強度も選べる（GoalKeeper準拠）。Forfeit のように罰金がアプリに入る構造ではなく、ウェルビーイング配慮を前面に出す。
4. **状況適応の分解**：締切・進捗・つまずき（F2）を踏まえて最初の一歩を出す。既存OSSの「タスク文だけから一律に分解」より文脈適応的。
5. **研究根拠の明示と単一ファイル完結**：CORE A以上の実証研究に紐づけて設計意図を説明でき、提出・デモが容易。
