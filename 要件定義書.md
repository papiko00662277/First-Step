# 要件定義書：「最初の一歩」タスク分解アプリ（FirstStep）

## 1. 背景・課題

やるべきことは分かっているのに、なかなか着手できず先延ばししてしまう。この原因の多くは「タスクが大きい・曖昧で、机に向かって"最初に何をすればいいか"が分からない」ことにある。

既存のToDoアプリは「タスクを記録する」ことはできても、「どう始めるか」までは助けてくれない。一方、生成AIに「このタスクを分解して」と単純に頼むと、「計画を立てる」「情報を集める」といった抽象的で一般論的なステップが返ってきて、結局その場で動けない。

つまり、**「曖昧で大きいタスク」と「今すぐ動ける具体行動」の間にあるギャップ**を埋める仕組みが必要。

## 2. 目的・ゴール

曖昧で大きな「やりたいこと」を、まず**達成までの大きな段階（フェーズ）**に分け、各段階から**今すぐ2分以内で着手できる具体的な「最初の一歩」**を切り出す。これにより着手のハードルを下げ、先延ばしを減らす。

「自分の生活を豊かにする」という観点では、先延ばしによる心理的負担の軽減と、小さな着手・振り返りの積み重ねによる達成感を狙う。

### 成功の定義（KPI 案）
- 提示された「最初の一歩」が、追加で考えずにそのまま行動に移せる割合
- 提示後に実際に着手できたタスクの割合（自己申告でも可）

## 3. ターゲットユーザー

自分自身（大学院生）。研究・授業課題・事務手続きなどで先延ばししがちな場面を想定。

### 想定シーン
- 「修論の章を書く」としか思いつかず、机に座っても手が止まる
- メール返信や書類提出など、面倒で後回しにしているタスクが溜まっている

## 4. スコープ

### 対象（In Scope）
- 「やりたいこと」の入力（締切の任意補足つき）
- 大きな段階（4〜6フェーズ）への分解
- 各段階からの「最初の一歩（2分）」のオンデマンド生成
- フォーカス実行画面（1つだけ全画面表示＋2分タイマー）
- 段階ごとの振り返りメモ（次段階の生成に反映）
- 複数の「やりたいこと」の管理（一覧・切替・削除）とブラウザ保存
- スマホ対応（ハンバーガー→ドロワー）

### 対象外（Out of Scope）
- 本格的なタスク管理（期限管理、リマインド通知、繰り返し設定）
- 複数ユーザー・共有・クラウド同期
- ペナルティ／コミットメント機構、ストリーク（「7. 検討したが不採用」を参照）

## 5. ユーザーストーリー

- user として、「レポートを書く」しか思いつかないとき、まず全体をいくつかの段階に分けたうえで、**机で最初にやる一手**を具体的に教えてほしい。動き出すきっかけが欲しいから。
- user として、分解された一歩が抽象的だと動けないので、**「動詞＋具体物」で2分以内に始められる粒度**で示してほしい。
- user として、締切や前の段階でのメモ（気づき・つまずき）を踏まえた最初の一歩にしてほしい。
- user として、複数の「やりたいこと」を並行して持てて、スマホでも画面を広く使いたい。

## 6. 機能要件

| ID | 機能 | 内容 |
|----|------|------|
| F1 | やりたいこと入力 | 自由記述で「やりたいこと」を入力できる |
| F2 | 文脈の補足（任意） | 締切を任意で入力できる。加えて各段階のメモが文脈として使われる |
| F3 | 段階分解 | 目標を着手順に4〜6の段階（フェーズ）へ分解する（大枠の見通し。細かい作業には刻まない） |
| F4 | 最初の一歩の生成 | 選んだ段階から、2分以内で着手可能な一歩を1つ、その場で生成する（動詞＋具体物＋小さな到達条件） |
| F5 | 自己検証 | 一歩が「説明文からズレていないか／2分で区切れるか／単一動作か／小さな痕跡が残るか」をAI自身が点検・修正してから出力する（プロンプト内） |
| F6 | フォーカス実行画面 | 最初の一歩1つだけを全画面表示＋2分カウントダウン。「できた ✓」「＋1分」で進める |
| F7 | 振り返りメモ | 各段階に「やったこと／分かったこと／困ったこと」のメモ欄。自動保存し、次以降の段階の最初の一歩生成に文脈として渡す（空の型だけの場合は渡さない） |
| F8 | 複数目標の管理 | サイドバーで「やりたいこと」を一覧・切替・削除。ブラウザ（localStorage）に保存 |
| F9 | スマホ対応 | スマホ幅ではサイドバーをハンバーガー（☰）から開くドロワーにし、画面を広く使う |
| F10 | 生成済みの一歩の保存 | 一度生成した最初の一歩は段階に保存し、再表示できる（開発用のリセットあり） |

## 7. 検討したが不採用（スコープ外）

当初は「着手の強制力」を高めるため、以下を検討したが、今回の実装スコープからは外した。設計判断として記録しておく。

- **ペナルティ／コミットメント機構（ロックアウト・掛け金）**：スキップ／未達時にアプリをロックする、または金銭・社会的コミットメントを課す仕組み。ウェルビーイングへの配慮と単一ファイルでの実装容易性を優先し、今回は見送り。
- **ストリーク（連続達成の記録）**：損失回避を継続に使う仕組み。上記と同様に見送り。
- **マルチプロバイダ対応（OpenAI互換／Anthropic等）**：今回は Google Gemini のみに一本化。
- **APIなしモード（プロンプトをコピーして手元のChatGP/Claudeに貼る）**：フローが二系統になり体験が分岐するため、今回は生成AIを直接呼ぶ方式に一本化。

なお、これらの背景となる先行研究（ペナルティ・コミットメント系）は「12. 理論的根拠」で参考として簡潔に触れるにとどめる。

## 8. 非機能要件

- **応答性**：入力から数秒〜十数秒で結果が返る。無料枠の毎分レート上限に合わせて呼び出しを制御し、超過時は自動リトライする。
- **わかりやすさ**：最初の一歩が一目で分かるUI（全画面のフォーカス表示）。
- **可搬性**：依存ライブラリなしの単一ファイル（`index.html`）で完結し、提出・デモ・GitHub Pages 公開が容易。
- **再現性**：同じ入力に対して品質が安定する（プロンプトで担保）。
- **プライバシー**：APIキーは既定でセッション内のみ保持。目標データは利用者の端末のブラウザ内のみに保存。

## 9. 入出力仕様

### 入力
- 必須：やりたいこと（自由記述）
- 任意：締切
- 生成時の文脈：前の段階のタイトル・説明・完了状態・メモ・提案済みの一歩（参考情報。実行済みとは断定しない）

### 出力
- **段階分解**：4〜6段階。各段階に `title`（段階名）と `note`（1行説明）。
- **最初の一歩**：選んだ段階から1つ。`step`（一歩）＋`nextHint`（終わった後の薄いヒント）＋`encouragement`（後押しの一言）。いずれもJSONで受け取り、アプリがフォーカスUI＋タイマーへ変換。

## 10. 工夫の方針（評価対象）

### 10-1. プロンプト設計の工夫 ―「単純プロンプトでは失敗した → だから○○を導入」

| 観察された失敗 | 原因 | 導入した工夫 |
|----------------|------|--------------|
| ステップが抽象的（「計画を立てる」） | 役割・粒度の指定がない | **役割付与**（先延ばし支援コーチ）＋**粒度制約**（「動詞＋具体物＋2分で区切れる単一動作」） |
| 対象を勝手に決め打ちする | 制約が緩い | **決め打ち防止**（入力にない対象・分野・教材・ツール・弱点・結果を推測させない） |
| 一般論で自分の状況に合わない | 良し悪しの基準が曖昧 | **良い例／悪い例（粒度の例）**をプロンプトに明示 |
| いきなり結論を出して粗い | 思考の段取りがない | **内部手順**（段階の作業列を推定→候補→自己点検→出力） |
| ユーザーの状況を無視 | 文脈が渡っていない | **文脈の差し込み**（締切・前段階・メモ）。ただし前の提案は実行結果と扱わない |
| たまに2分で始められない一手を出す | 出力後の自己点検がない | **自己検証ステップ**（self-check）で再評価・修正させる |

### 10-2. 設計プロセスの工夫 ―「なぜその手順か」を説明できるように

- **なぜこの手順か**：要件定義 → プロンプト試作 → 失敗の観察（実タスクで検証） → 改善の反復 → UI実装、という順で進める。先にプロンプトの良し悪しを確かめてからUIに固めることで、作り直しを減らす。
- **なぜこの構成か**：段階（全体像）と「最初の一歩」を分け、着手時は一歩だけを主役に据える。分解結果を全部見せるより、今やる一つを強調する方が着手率が上がるという仮説に基づく。

## 11. 実装形態

Webアプリ（HTML/JS 単体、依存ライブラリなし）。生成AIは **Google Gemini API** をブラウザから直接呼び出す。APIキーは既定で `sessionStorage`、「この端末に保存する」を選んだ場合のみ `localStorage` に保存。目標データ・メモは `localStorage` に保存。GitHub Pages でそのまま公開可能。

## 12. 理論的根拠（先行研究レビュー）

本アプリのコンセプト「**最初の一歩（最初の数分）を具体化することが、着手のカギになる**」を、行動変容・HCI 分野の先行研究で裏づける。

**採録基準**：参照する論文は、すべて **CORE Rank A 以上**（A／A*）の会議に採録されたものに限定し、できるだけ新しいものを優先する。CORE は計算機科学系の会議を対象とするランキングのため、本レビューでは行動変容・HCI のトップ会議（CHI＝A*）の実証研究を採用する。心理学・経済学のジャーナル論文は概念的背景としては有名だが CORE の対象外のため、参照文献からは外している。

### テーマ：曖昧なタスクを「分解」して「最初の一歩」で着手させる

#### 12-1. 大きなタスクは「分解」した方が、質が上がり中断にも強い ＜CHI 2015 / A*＞
Cheng, Teevan, Iqbal & Bernstein (2015, CHI) は、まとまった作業（マクロタスク）を小さな単位（マイクロタスク）に分解する効果を実験的に検証し、分解すると**成果物の質が上がり、中断からの復帰に強く、体験も良くなる**ことを示した。

→ **アプリへの反映**：曖昧で大きい目標を段階に分解する中核機能（F3）と、特に着手単位を最小化する設計（F4）の根拠。

#### 12-2. 生成AIが「停滞」を検知し、続きを見せて再開のハードルを下げる ＜CHI 2023 / A*＞
Arakawa, Yakura & Goto (2023, CHI) の CatAlyst は、作業の停滞を検知し、**大規模生成モデルが「作業の続き」を生成して提示する**ことで、先延ばしを防ぎ再開のハードルを下げる介入を提案・評価した。成果物を肩代わりするのではなく、関心を引いて着手・再開を促す点が要点。

→ **アプリへの反映**：生成AIで「最初の一歩」を具体的に示して着手のハードルを下げる本アプリの中核アプローチ（F4／F6）を、最新のHCI研究が直接裏づける。

### 参考：ペナルティ・コミットメント（今回は不採用の背景）
初期構想ではペナルティ機構（F7旧案）を検討し、その根拠として Lee et al.（2021, CHI, Sticky Goals）や Kim et al.（2019, IMWUT, GoalKeeper）を参照していた。今回の実装ではこの機構をスコープ外としたため、これらは背景文献の位置づけにとどめる（詳細は「7. 検討したが不採用」）。

### 注記：「2分ルール」など実務フレームワークとの関係
着想のもとになった「2分ルール」は David Allen『Getting Things Done』や James Clear『Atomic Habits』で広まった**実務フレームワーク**であり、それ自体は査読論文ではない。本書では効果の根拠を上記の CORE A 以上の実証研究に置き、実務ルールは説明の補助としてのみ扱う。

## 14. 確定した設計判断（旧「未確定事項」）

## 15. 競合・類似アプリ調査と差別化方針

GitHub・各種プロダクトを調査した結果、類似アプリは大きく2系統（AIで着手支援する系／お金を賭けるペナルティ系）に分かれる。本アプリは前者に位置づけつつ、**段階と一歩の分離＋振り返りメモの引き継ぎ**という設計で差別化する。

### 15-1. 系統A：AIで「最初の一歩」に分解する着手支援系

| アプリ | 概要 | 強み | 本アプリの余地 |
|--------|------|------|--------------------------|
| **Thawly.ai** | タスクを「最小の身体的マイクロ行動」へ即分解し、2分タイマーで勢いをつける | 最小行動＋2分の組み合わせ | 段階（全体像）と一歩を分けていない。振り返りメモの文脈引き継ぎがない |
| **ADHD Guardian**（OSS） | GPT-4oでタスクを3〜5個の一歩に分解＋非批判的な励まし | OSSで実装参照可 | 段階分解が浅い。前の気づきを次の生成へ渡さない。デモ段階 |
| **adhd-tasker**（OSS） | 「今やる1タスクだけ全画面表示」 | 集中UIが明快 | AI分解なし |
| Shrimp Task Manager 等（OSS） | 開発者向けのタスク自動分解 | 依存関係管理が強力 | 対象がソフトウェア開発で、個人の生活・着手心理は対象外 |

### 15-2. 系統B：お金を賭けるペナルティ系（本アプリは非採用）

stickK・Beeminder・Forfeit などは強い継続力を持つが、「何をやるか」が既に分かっている前提で、**大きく曖昧なタスクを最初の一歩に砕く部分を助けない**。本アプリはこの系統の強制力は採らず、着手ハードルの最小化に集中する（ペナルティを外した理由は「7」）。

### 15-3. 本アプリの独自化方針

1. **段階と「最初の一歩」の分離**：全体像（4〜6段階）と、その場で生成する2分の一歩を分けて扱う。着手時は一歩だけを主役にし、計画作業に引き戻さない。
2. **振り返りメモの文脈引き継ぎ**：各段階のメモ（やったこと／分かったこと／困ったこと）を、次以降の段階の一歩生成に渡す。進むほど文脈が積み上がる。
3. **AIの自己検証ループ**：一歩が「2分で区切れる単一動作か／決め打ちしていないか／痕跡が残るか」をAI自身が点検・修正してから提示（F5）。
4. **決め打ち防止と状況適応**：入力にない対象を推測させず、締切・前段階・メモを踏まえる。
5. **研究根拠の明示と単一ファイル完結**：CORE A以上の実証研究に紐づけて設計意図を説明でき、提出・デモが容易。
